# AI 知识库 · 编码规范 v1.0

## 1. 适用范围

本规范适用于本项目内所有 Python 代码、配置脚本、自动化脚本以及与 AI 知识库流水线相关的新增实现。

## 2. 代码风格

- Python 版本统一使用 Python 3.11
- 遵循 PEP 8 规范
- 使用 `black` 进行代码格式化，格式化结果为唯一标准
- 使用 `ruff` 进行静态检查与基础代码质量约束
- 变量名、函数名、模块名统一使用 `snake_case`
- 类名统一使用 `PascalCase`
- 常量统一使用 `UPPER_SNAKE_CASE`
- 文件编码统一为 UTF-8

## 3. 文档规范

- 所有公开函数、公开类和公开模块都必须提供 docstring
- docstring 统一使用 Google 风格
- docstring 必须说明用途、参数、返回值；如有异常抛出，也必须说明
- 对于复杂逻辑的内部函数，如影响可读性或承载关键业务逻辑，也应补充简短说明

## 4. 日志与输出规范

- 禁止使用裸 `print()`
- 统一使用标准日志库 `logging`
- 日志应按场景选择合适级别：`DEBUG`、`INFO`、`WARNING`、`ERROR`、`CRITICAL`
- 日志内容不得包含 API Key、Token、Cookie、密码、隐私数据等敏感信息
- 对外输出或持久化结果优先写入文件、JSON 或结构化日志，而不是直接打印到控制台

## 5. 常量与字符串规范

- 禁止散落硬编码的“魔法字符串”
- 以下情况允许直接使用字符串字面量：
  - API 路径、协议字段名、JSON 键名等外部接口约定
  - 文案、提示词、文件名、状态值等有明确业务语义且复用价值不高的场景
- 对于在多个位置复用、且一旦修改会影响多个模块的值，应提取为命名常量
- 不因“避免魔法字符串”而牺牲可读性

## 6. 类型与结构约束

- 优先使用清晰、可读、可维护的函数签名
- 新增公共函数应优先补充类型注解
- 不得使用 `import *`
- 单个函数应尽量保持职责单一，避免过长、过深嵌套和过度分支
- 业务配置、环境变量、敏感参数不得写死在代码中

## 7. 测试规范

- 新增或修改核心逻辑时必须补充测试
- 单元测试优先使用 `pytest`
- 新增功能的测试应覆盖正常路径、边界情况和失败路径
- 项目核心模块的单测覆盖率目标为 80% 以上
- 覆盖率以 CI 中的统一口径为准，优先关注关键业务路径，而不是机械凑数

## 8. 提交前检查

提交前至少完成以下检查：

- 代码格式化通过
- 静态检查通过
- 相关测试通过
- 不存在未解释的 TODO、临时调试代码或明显残留日志

## 9. 验证标准

CI 至少应执行以下检查：

- `black` 格式检查
- `ruff` 静态检查
- `pytest` 单元测试

如项目后续引入类型检查、集成测试或安全扫描，应在 CI 中同步补充。

## 10. 红线（绝对禁止）

- 禁止提交包含密钥、Token、Cookie、密码等敏感信息的代码或配置
- 禁止未经验证就将错误数据写入知识库或分发渠道
- 禁止绕过测试、静态检查或审查流程直接合并到主分支
- 禁止在生产逻辑中保留裸 `print()`、临时调试输出或无意义占位实现
- 禁止提交未说明用途的 TODO、FIXME 或临时代码
- 禁止随意删除知识数据、测试数据或审计痕迹
